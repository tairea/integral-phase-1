# DR-001 — Adopt the Phase-1 candidate schema set as working baseline & open ratification

> **Status of this record.** This is a **candidate / worked instance** — the first entry in the
> `integral/decisions` append-only record (DEVGUIDE §6.2), produced by *using the CDS schema this
> exercise defined to govern the exercise's own ratification* (OQ-12 dogfooding). It is **not** an
> enacted governance decision: in Phase 1 the contributor working group does not yet exist
> (applications are still being processed — Ep.59). The working-group votes below are **illustrative**
> (`votes.json → illustrative: true`), standing in for the deliberation the real WG / trustee will
> run. What this decision legitimately records is a **development-process** decision (§6.3, the
> "provisional precursor to CDS"): *adopt a review baseline and open the questions* — it does **not**
> finalize the schema. The substantive ratification (the 16 OQs) remains open.

## Why this exists

Two reasons, both load-bearing:

1. **Dogfooding (OQ-12).** If the CDS schema can faithfully represent its own ratification, that is
   the strongest evidence it works. Building DR-001 already paid off: it surfaced a real gap —
   `DecisionRecord` was missing `governance_scope` (the cross-contract matrix claimed it had it).
   Fixed before instantiating. Instantiation *is* validation.
2. **It opens the ratification process** §5.1 requires before Phase 2.

## The decision

**Approved:** Scenario **SCN-001** — *Adopt candidate schema set v0.1 as the working baseline; open
all 16 OQs for community ratification; gate Phase 2 on OQ-01 (notation/source-of-truth) and OQ-04
(decay policy/snapshot sync).*

`governance_scope: node` · `decided_at: 2026-05-30` · append-only chain start (`prev_hash: GENESIS`).

### Module 6 result (consensus math, computed not asserted)

Using the canonical asymmetric gradient (Ep.59 frame 65: `strong_support=1.0, support=0.6,
neutral=0.0, concern=-0.4, block=-1.0`) and node thresholds `consensus=0.72 / block=0.30`:

| Scenario | consensus_score | objection_index | directive |
|----------|----------------:|----------------:|-----------|
| **SCN-001** adopt as baseline + open ratification | **+0.8667** | **0.09** | **approve** ✅ |
| SCN-002 ratify as FINAL now | +0.6333 | **0.56** | revise ❌ |
| SCN-003 defer entirely | −0.1000 | 0.00 | revise ❌ |

**The objection mechanism did the real work.** SCN-002 ("ratify as final now") was killed not by
unpopularity (it had decent support) but by **OBJ-001** (severity 0.8 × scope 0.7 = 0.56 ≥ 0.30):
ratifying while OQ-01 and OQ-04 are open risks exactly the unfixable schema-migration / historical-
record invalidation DEVGUIDE §4.1 warns about. This is the white paper's "a proposal can be popular
and still be blocked by one well-grounded objection" asymmetry, working as designed.

## What the dispatch sets in motion (DSP-001)

- **Tasks** → `specifications-wg`: open OQ-01..OQ-16 ratification sessions; per-system WGs review
  specs 01–05; joint `#specifications` resolves cross-cutting OQ-01/04/07/08/16.
- **FRS monitor / gate** → `Phase-2 gate: OQ-01 AND OQ-04 must close before Phase 2 begins.`
- **Review triggers** recorded on DR-001 (`new_evidence`, `changed_conditions`) so that if the OQ
  resolutions materially change the schema, Module 10 has the hooks to reopen this.

## Reproduce / verify

```bash
python3 decisions/DR-001-adopt-schema-baseline/build_and_validate.py
# → PASS Issue / DecisionRecord / DispatchPacket; prints the consensus table + real hashes
```

The script constructs every instance, computes the consensus math and sha256 hashes, writes
`instances/*.json`, and validates them against `schemas/`. Deterministic (timestamp is passed in,
not generated), so re-running yields identical hashes.

## What DR-001 deliberately does NOT do

- It does **not** ratify the schema as final (that needs the real WG closing the OQs).
- It does **not** open Phase 2 (gated on OQ-01 + OQ-04).
- The votes are **not** real consensus — they are flagged illustrative until the WG exists.
