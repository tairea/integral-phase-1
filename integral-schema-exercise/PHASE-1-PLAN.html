Integral · Phase 1 · A proposal for the working groups

Get the foundation
right, once.

Before anyone writes production code, we agree on the shape of Integral's data — the schema. This page explains why the Dev Guide treats that as the single highest-leverage job in the project, what questions we still need to answer together, and who's doing what.

A plain-language walkthrough 9 workstreams 17 open questions 2 must close before Phase 2
// Why this work, and why now

This is pouring the foundation

Imagine building a house. You could start framing walls tomorrow — or spend a few extra days making sure every measurement on the foundation is right. With walls, a mistake is annoying to fix. With the foundation, it's not.

The schema is Integral's foundation. It's the agreed shape of every record the system keeps — a decision, a unit of work, a credit, a design, a signal. The Dev Guide is unusually blunt about how much this matters:

"This is not glamorous work and it does not produce running software — but it is arguably the highest-leverage technical work in the entire project." Integral Dev Guide · §5.1

The reason is simple. Integral's records live in append-only ledgers — a permanent history that's never rewritten. That's a feature: it's how the system stays honest and accountable. But it also means that whatever shape we choose, every record ever written is stuck in it.

Do the exercise first

Every team builds against one complete, agreed shape. The five systems connect cleanly the first time, and the records we write today still make sense years from now.

Skip it and build fast

A quick local win — then a permanent problem. The ledgers fill with records in a shape that doesn't fit, and that mismatch is carried forever or fixed at very high cost. This exercise is the insurance against that.

📖 Read the actual Dev Guide passage — right here +

"Before Phase 2 development begins, a deliberate schema design exercise must be completed… Working group by working group, contributors sit with the white paper's full module specifications and the data contracts in Section 4, and ask of every object and every field: will the full system eventually need this, and if so, is it represented in the schema?

The output of this exercise is a ratified, complete schema for each system — with all deferred fields present as typed, nullable placeholders — before the first line of production code is written. Teams that skip this step and build quickly against an incomplete schema will produce a local success and a future migration problem. The append-only ledgers that form the system's institutional memory will contain records in an incompatible format. That is not a bug that gets fixed — it is a constraint that gets carried indefinitely or resolved at very high cost. The schema design exercise is the insurance against that outcome. Its completion is a condition for Phase 2 to begin."

Dev Guide §5.1 · view on GitHub ↗
📖 …and why coupling makes this critical (Dev Guide §4.1) +

"The five systems of Integral are not independent applications that happen to share a network. They are deeply coupled components of a single architecture… If COS produces labor event records in a format that ITC cannot reliably consume, contribution accounting breaks. If ITC's credit records do not carry the fields that FRS needs to assess equity patterns, the feedback loop is blind to fairness drift.

These are not bugs that can be fixed later with a patch. They are structural incompatibilities that require schema migrations, data backfilling, and potentially the invalidation of historical records — exactly the kind of disruption that destroys institutional memory and erodes contributor trust in a young project."

// Before we go further — a fair question

"Hasn't this already been decided, then?"

Short answer: no. Nothing here is adopted.

A first candidate draft of the schema already exists — but it was built precisely so the community has something concrete to react to, not a blank page to stare at. It's far easier to challenge a real proposal than to design one from scratch in a meeting.

Every choice in it is labelled as a proposal, and the hardest questions were deliberately left open — that's exactly what this Phase 1 round is for. In the words of the draft itself: it is "offered to the collective to challenge, refine, and ratify." The teams below are how we do that, together.

// What we're actually deciding

So what are these "open questions"?

The Dev Guide gives a clear instruction: sit with the full specifications and, for every object and every field, ask one question —

"Will the full system eventually need this, and if so, is it represented in the schema?"Integral Dev Guide · §5.1

We did exactly that, against the candidate draft. The exercise surfaced 17 genuine open questions — places where the source documents leave a real choice, or quietly disagree with each other. Each comes with a proposed direction, but none are decided. Deciding them together is the work.

Two questions are marked 🔴 must-close: the system genuinely cannot begin Phase 2 until they're settled. The rest can be resolved alongside the build or safely deferred — each is labelled so you can see which is which.

// How the work flows

Foundation first, then everyone in parallel

One team agrees the ground rules. Then the five system teams work side by side. Two small review teams check the whole set, and a final session ratifies it — the moment Phase 2 begins.

OPEN-QUESTION-01Which format is the "official" schema written in?+
In plain terms
We choose one format that a computer can check automatically, so every team builds against the exact same definition — not their own reading of a document.
Why it matters
This decides how every other piece of work is written and reviewed. Until it's set, two teams can read the same spec and build incompatible things.
Directions on the table
Proposed: keep JSON Schema as the source of truth (checkable, automatable), with readable docs kept beside it; leave language-specific versions to Phase 2.
Worth weighing: is machine-checking worth a small learning curve, versus plain annotated documents anyone can edit?

Questions this team owns
OPEN-QUESTION-17There's no way to file a simple "problem report"+
In plain terms
The most common thing a member does is report a problem — "the footbridge floods." But the current list of submission types (proposal, objection, evidence, comment, signal) has no slot for that.
Why it matters
If a plain problem report has no home, every community invents its own label and the data fragments — exactly what a shared schema exists to prevent. Our test simulator hit this immediately.
Directions on the table
Proposed: add "concern" as a sixth submission type.
Worth weighing: folding problem reports into "comment" loses the "this needs resolving" signal — and treating every report as a "proposal" overstates intent (a report isn't yet a proposed action).
Source
White Paper — CDS Module 1; surfaced by the data-flow simulator
OPEN-QUESTION-14Should every vote count equally?+
In plain terms
Normally one person = one vote. The question is whether expert input should ever weigh more in narrow cases.
Why it matters
This is the equality-vs-expertise tension made concrete. Adding a weighting column to a permanent ledger later is very costly — so we reserve the space now even if we never use it.
Directions on the table
Proposed: keep the field, default it to equal (1.0), and lock any unequal value behind a policy the community must ratify.
Worth weighing: whether to allow unequal voting at all is a values question, not a technical one — member input wanted.
Source
White Paper — CDS Module 6; Ep.59 discussion
OPEN-QUESTION-15Where do the "consensus reached" numbers live?+
In plain terms
The thresholds that decide "we've reached agreement" — should they be settings each community can tune, or fixed in code?
Why it matters
Lets each node adapt its own governance and lets the wider network learn what works.
Directions on the table
Proposed: the numbers live in a tunable policy record per node; the agreement labels (e.g. "strong support") stay fixed and shared.
Worth weighing: if every node tunes freely, governance styles could become hard to compare across the network — what should stay standard?
Source
White Paper consensus model; Ep.59 (1:00:45)
OPEN-QUESTION-13Do co-ops get their own decision system, or share the node's?+
In plain terms
Co-ops run a smaller version of the decision process and escalate bigger matters up to the node. Same schema, or a separate one?
Why it matters
Affects every community with more than one co-op (Phase 3+). Keeping records uniform across levels makes escalation simple.
Directions on the table
Proposed: one schema, tagged with a "scope" and a record of what it escalated from.
Worth weighing: a distinct sub-schema for co-ops gives cleaner separation, but means two things to maintain and map between.
Source
Ep.59 (30:37)
OPEN-QUESTION-06Should a decision remember which community it came from?+
In plain terms
For a future network of many communities, add an optional field noting a decision's origin. It stays empty while we're a single community.
Why it matters
Cheap to add now, painful to retrofit into a permanent ledger later.
Directions on the table
Proposed: add the optional origin field now, leave it null in single-community use.
Worth weighing: is a single "origin node" enough, or will cross-community decisions later need richer provenance (who referenced or ratified it)?
Source

WORKSTREAM-2 · ITC — CREDITS🔴 contains a must-close

How contribution is credited & how credits fade

Lock the shape of a member's credit account and the ledger that records every issue and use of credit. Credits in Integral aren't money — they fade over time and disappear when used, so the rules here protect a core principle.

▶Walk through the ITCSee credits issued for footbridge work, then extinguished on access↗
LFLead Lynn Foster LELeanne BNBen VTVerta PJPeter Joseph
Questions this team owns
OPEN-QUESTION-04Keeping the "decay" setting from drifting apart+
In plain terms
Credits fade over time so they can't pile up like money. That fade setting is stored in two places. We need a rule so the two can never disagree.
Why it matters
This protects the promise that credits don't accumulate — one of Integral's foundational principles. If the two copies drift, the math behind the whole credit system becomes untrustworthy.
Directions on the table
Proposed: the account stores only the rule's ID; the inline numbers are a read-only copy refreshed automatically each time decay is applied, and never edited directly.
Worth weighing: the master copy lives in governance (CDS) — we need to confirm the refresh timing so an account never decays using stale settings.
Source
Dev Guide §4.2 — CDS→ITC Policy Signal ↗ · White Paper §1.3
📖 Read the passage here +

"CDS → ITC: The Policy Signal — The contract by which CDS governance decisions become the normative rules that ITC operates within. This includes labor weighting parameters, decay rules, fairness thresholds, access policy modifications, and anti-coercion boundaries. ITC does not set its own policy — CDS does."

Dev Guide §4.2
OPEN-QUESTION-03Keep "raw work done" and "credited value" as one record or two?+
In plain terms
When work is recorded there's the raw fact ("X did 3 hours") and the credited value derived from it. One record or two?
Why it matters
The raw record is meant to be value-neutral. If we ever re-credit work under a new fairness policy, we must not rewrite what actually happened.
Directions on the table
Proposed: merge into one record, but freeze the raw part once written and version the credit part separately. Ratify the freeze rule now.
Worth weighing: if a future policy re-weights old work, does versioning the credit block fully preserve the original — or do we ultimately need two separate records?
Source
Dev Guide §4.3 ↗
📖 Read the passage here +

"The white paper separates raw labor capture (LaborEvent) from weighted contribution recognition (WeightedLaborRecord). This is architecturally significant: the labor event is value-neutral until weighting is applied… the boundary between raw capture and weighted output must be preserved in the field structure."

OPEN-QUESTION-07One shared scale for "how serious is this?"+
In plain terms
Different systems currently rate seriousness on different scales (3 levels vs 5).
Why it matters
The feedback system needs to compare issues across systems — impossible if each one uses its own scale.
Directions on the table
Proposed: one shared 5-level scale, mapping the credit system's "warning" onto "moderate."
Worth weighing: ethics violations may deserve their own scale — e.g. a hard "critical" floor for coercion. Practitioner input wanted.
Source
White Paper — ITC / FRS / CDS severity definitions
OPEN-QUESTION-08One shared word for "spending" a credit+
In plain terms
Two systems describe the same event — a credit being used up to access something.
Why it matters
This is the heart of the "credits are extinguished on access" loop — the moment a credit does its job and disappears.
Directions on the table
Proposed: confirm both systems use one shared "access mode" vocabulary — two views of the same event that must never diverge.
Worth weighing: is there ever a legitimate case where "spent" and "accessed" should differ? If not, lock them as one.
Source
White Paper — ITC redemption / COS distribution

OPEN-QUESTION-02Save a design's "value signal," or recalculate it each time?+
In plain terms
When a certified design hands a condensed "value signal" to the credit system, do we save a snapshot or recompute it every time it's read?
Why it matters
Stable, auditable credit values you can trace back — versus numbers that silently change underneath you.
Directions on the table
Proposed: save a snapshot, stamped with the design version and policy that produced it.
Worth weighing: computing on read is always fresh, but costs recompute and keeps no historical snapshot to audit against.

OPEN-QUESTION-05Make design history machine-readable+
In plain terms
A design's change history is currently free-form text for humans. The feedback system needs it as structured events it can read.
Why it matters
Enables the feedback system to learn from how designs evolve and get re-certified.
Directions on the table
Proposed: add structured design + certification-change events as first-class records, and keep the free-text changelog as the human-readable narrative alongside them.
Worth weighing: how much history is worth capturing as structured events versus left as narrative — where's the sensible line?

OPEN-QUESTION-09Record where materials came from — in detail+
In plain terms
Store the detailed source of materials (recycled / brought in from outside / used in production / scrapped), and derive the simpler version whenever a connecting system needs it.
Why it matters
No information is lost, and the feedback system gets recycling and waste signals "for free" — useful for ecological awareness.
Directions on the table
Proposed: store the richer 4-way source value, derive the coarse 2-way value for the credit contract.
Worth weighing: are four categories enough, or will some nodes need finer detail later (e.g. repair vs reuse)?

How the system spots problems & advises

▶Walk through the FRSWatch operational signals become a finding fed back to governance↗
AXLead Alex DVDavid VIVitali
Questions this team owns
OPEN-QUESTION-11The feedback system advises — it must never decide+
In plain terms
The feedback system flags issues and recommends fixes — but it must be structurally impossible for it to approve its own recommendation.
Why it matters
Prevents "shadow governance" — a monitoring system quietly becoming the decision-maker. The White Paper calls this boundary structural, not just a rule people are asked to follow.
Directions on the table
Proposed: make the "accepted" status writable only by the target system or the community — never by the feedback system — enforced in the software, not just by convention.
Worth weighing: exactly where that lock is enforced, and how we test that the feedback system truly cannot write "accepted."
Source
White Paper §2.1 (advisory-only guarantee)
OPEN-QUESTION-16Two kinds of "scope" that look the same but aren't+
In plain terms
"How far a problem reaches" and "who has the authority to act on it" are genuinely different things.
Why it matters
Gets escalation right once co-ops and multi-community networks exist — a node-wide problem can still be handled at co-op level if it's contained there.
Directions on the table
Proposed: keep them as two clearly-labelled ideas with a defined map between them.
Worth weighing: a single unified scale removes the map, but blurs "reach" and "authority," which really do differ.
Source
White Paper — FRS scope; Ep.59

Also weighs in on: OPEN-QUESTION-05 (with OAD) and OPEN-QUESTION-07 (with ITC).

Decide together, write it down, begin building

OPEN-QUESTION-10What about the parts we haven't fully formalized yet?+
In plain terms
This first pass fully formalizes only the records that pass between systems (about 15 of roughly 80). What about the internal-only ones?
Why it matters
Honesty about coverage. Everyone can see what's done and what's still coming, so there are no hidden gaps that surprise us later.
Directions on the table
Proposed: each team formalizes its internal-only records as that part of the system matures, checked against the shared lists ratified now — openly tracked, never quietly dropped.
Worth weighing: who decides when a deferred record is "mature enough" to formalize, so it doesn't slip indefinitely?
